# C3 — An Evolution of C (Experimental)

## Overview

C3 is an experimental systems programming language created by Christoffer Lerno, designed
as a modernized evolution of C. Rather than replacing C with a fundamentally different
paradigm (like Rust or Zig), C3 keeps C's model — manual memory, procedural style,
close to the hardware — while fixing its worst problems: undefined behavior, lack of
generics, dangerous macros, and missing error handling.

- **Current version**: 0.7.5 (2025) — **pre-1.0, experimental**
- **Paradigm**: Imperative, procedural, generic
- **Typing**: Static, strong
- **Compilation**: AOT via LLVM backend
- **Package manager**: Built-in project system (no central registry yet)
- **GC**: None — manual memory management
- **License**: MIT
- **Status**: Active development, sole developer (Christoffer Lerno, spare-time project)

> **Warning**: C3 is pre-1.0 and experimental. APIs may change between versions. The
> standard library is less mature than the compiler. The ecosystem is minimal. Consider
> it for exploration and experimentation, not production workloads.

### Design Philosophy
"An evolution, not a revolution." C3 should feel familiar to any C programmer — same
mental model, same performance characteristics, same interop story — but with the sharp
edges filed down. No header files, no preprocessor, no undefined behavior surprises,
proper generics, real error handling, and contracts.

## Reduced Undefined Behavior

C3 defines behavior where C leaves it undefined:

- **Signed integer overflow**: Defined as two's complement wrapping (not UB)
- **Shift amounts**: Defined behavior for over-shifting
- **Null pointer dereference**: Trapped, not UB
- **Uninitialized variables**: Zero-initialized by default (opt-out with `@noinit`)
- **Union access**: Defined behavior for type-punning through unions
- **Pointer arithmetic**: Bounds-checked in safe mode

```c3
int x = int.max;
x += 1;  // Defined: wraps to int.min (not UB like in C)

int[4] arr;  // Zero-initialized by default
// arr[5];   // Trapped in safe mode — not silent UB
```

The goal is that a C3 program's behavior should always be explainable by looking at the
code. No "nasal demons."

## Contracts

C3 implements design-by-contract with `@require`, `@ensure`, and `@pure` annotations.
Contracts are checked at compile time where possible and at runtime otherwise.

```c3
fn int safe_divide(int a, int b)
    @require(b != 0, "Division by zero")
    @ensure(result => a == b * result + a % b)
{
    return a / b;
}

fn void insert(List* list, int index, int value)
    @require(index >= 0 && index <= list.len, "Index out of bounds")
    @ensure(list.len == old(list.len) + 1)
{
    // ...
}
```

- `@require` — preconditions on function arguments
- `@ensure` — postconditions on return values (access via `result`)
- `old()` — captures pre-call state for postcondition comparisons
- `@pure` — function has no side effects
- Contracts can be stripped in release builds (like D's `in`/`out` contracts)

## Generic Modules

C3's generics use parameterized modules rather than templates or monomorphized generics.
This avoids both C's macro-based "generics" and C++ template error novels.

```c3
module stack(<Type>);

struct Stack {
    Type[] data;
    usz count;
}

fn void Stack.push(Stack* s, Type value) {
    s.data[s.count++] = value;
}

fn Type Stack.pop(Stack* s) @require(s.count > 0) {
    return s.data[--s.count];
}

// Instantiation
def IntStack = stack(<int>).Stack;
def FloatStack = stack(<float>).Stack;
```

- Type parameters at the module level, not per-function
- Clean error messages — no deeply nested template errors
- Contracts carry over to generic instantiations
- Standard library uses generic modules for containers (`List`, `HashMap`, etc.)

## Semantic Macros

C3 replaces C's textual `#define` with semantic macros that understand types, scope, and
the AST. They look like function calls and behave predictably.

```c3
// Macro that works like a function — no textual surprises
macro int max(int a, int b) {
    return a > b ? a : b;
}

// Macros can take code blocks
macro void @timed(#body) {
    Clock start = clock::now();
    #body;
    io::printfn("Elapsed: %s ms", clock::now() - start);
}

fn void process() {
    @timed({
        heavy_computation();
    });
}

// Compile-time evaluation
macro int @ct_factorial($n) {
    $if $n <= 1:
        return 1;
    $else:
        return $n * @ct_factorial($n - 1);
    $endif;
}
```

- No textual substitution — macros respect types, scoping, hygiene
- `#body` parameters for code block injection
- `$if`, `$for`, `$switch` for compile-time control flow
- `$typeof`, `$sizeof` for compile-time reflection
- Vastly safer than C preprocessor macros

## Opt-In Safety

C3 provides safety features that can be enabled or disabled per-scope:

```c3
// Safe mode: bounds checking, null checks, overflow traps
fn void safe_function() @safe {
    int[4] arr;
    // arr[10] = 5;  // Runtime trap — bounds violation
}

// Default mode: checks in debug, stripped in release
fn void normal_function() {
    // Standard safety checks active in debug builds
}
```

- Bounds checking on arrays and slices
- Null safety via optional types (`Type?` / `Type!`)
- Integer overflow detection
- Safe/unsafe annotations for granular control
- Debug builds have full checks; release builds strip them

## Error Handling

C3 uses optional/result types integrated into the language, similar to Zig and Rust but
with C-like ergonomics.

```c3
// Functions that can fail return Type! (result type)
fn int! parse_int(char[] str) {
    if (str.len == 0) return InvalidArgument?;
    // ... parsing logic ...
    return value;
}

// Propagate with ! (like Rust's ? or Zig's try)
fn int! load_config() {
    int port = parse_int(read_file("config.txt")!)!;
    return port;
}

// Handle errors explicitly
fn void main() {
    if (catch err = load_config()) {
        io::printfn("Config error: %s", err);
        return;
    }
    // Or with default
    int port = load_config() ?? 8080;
}
```

- `Type!` — result type (value or fault)
- `!` suffix propagates errors up
- `??` for default values on error
- `if (catch err = ...)` for explicit error handling
- No exceptions, no hidden control flow
- Named faults instead of error codes

## Type System

```c3
// Distinct types — not just aliases, truly separate types
def Meters = distinct float;
def Seconds = distinct float;

fn Meters distance(Meters m, Seconds s) {
    // return m + s;  // Compile error — different types!
    return m;
}

// Enum with associated data (basic sum types)
enum Color : int {
    RED,
    GREEN,
    BLUE,
    CUSTOM = 99,
}

// Bitstruct — typed bitfield access
bitstruct Flags : char {
    bool readable : 0;
    bool writable : 1;
    bool executable : 2;
}

// Interfaces via structural typing
interface Printable {
    fn void print(void* self);
}
```

- Distinct types prevent unit confusion (meters vs seconds)
- `bitstruct` for hardware register / protocol definitions
- Slices with length (`int[]`) instead of raw pointers
- Built-in optional types (`Type?`)
- No implicit type conversions

## C Interop

C3 has seamless C interop — no FFI layer, no bindings generation needed.

```c3
// Import C headers directly
extern fn int printf(char* fmt, ...);

// Link C libraries
module myapp;
import libc;

fn void main() {
    libc::printf("Hello from C3\n");
}
```

- C ABI compatible by default
- Can call C functions directly
- C libraries link without wrappers
- Existing C codebases can adopt C3 incrementally

## Build & Targets

```bash
c3c init myproject        # create project
c3c build                 # build
c3c run                   # build and run
c3c test                  # run tests
c3c build --target linux-x64    # cross-compile
```

- LLVM backend — targets whatever LLVM targets (x86, ARM, WASM, RISC-V)
- Cross-compilation support
- Built-in project system with `project.json`
- No external build tools needed
- Standard library focus for 0.8-0.9 releases

## Comparison with C

| Aspect | C3 | C |
|---|---|---|
| **Undefined behavior** | Minimal, defined semantics | Pervasive |
| **Generics** | Generic modules | Preprocessor macros |
| **Macros** | Semantic (type-aware) | Textual (#define) |
| **Error handling** | Result types (Type!) | Return codes |
| **Contracts** | Built-in (@require/@ensure) | None (assert only) |
| **Headers** | No header files | Header/source split |
| **Arrays** | Slices with length | Raw pointers + size |
| **Initialization** | Zero by default | Uninitialized (UB) |
| **Interop** | Direct C ABI | N/A |

## Comparison with Zig

| Aspect | C3 | Zig |
|---|---|---|
| **Philosophy** | Evolve C conservatively | Replace C fundamentally |
| **Metaprogramming** | Semantic macros | comptime (more powerful) |
| **Allocators** | Traditional (malloc-style) | Explicit allocator parameters |
| **Maturity** | Pre-1.0, one developer | Pre-1.0, funded team + community |
| **Ecosystem** | Very small | Small but growing |
| **Contracts** | Built-in | Not built-in |
| **C interop** | Direct C ABI | @cImport (reads C headers) |
| **Backend** | LLVM only | LLVM + self-hosted |

## Strengths

- **Familiar to C programmers** — minimal conceptual overhead to adopt
- **Contracts** — design-by-contract built into the language
- **Semantic macros** — vastly safer than C preprocessor
- **Generic modules** — clean generics without template complexity
- **Reduced UB** — defined semantics for overflow, shifts, initialization
- **C interop** — drop-in replacement for C code, same ABI
- **Opt-in safety** — bounds checking, null safety without all-or-nothing

## Weaknesses

- **Experimental** — pre-1.0, API instability, no stability guarantees
- **One developer** — bus factor of 1; sustainability risk
- **Tiny ecosystem** — no central package registry, very few third-party libraries
- **Immature stdlib** — standard library acknowledged as needing more work
- **No production track record** — no known large-scale production deployments
- **LLVM-only** — no alternative backends for fast debug compilation
- **Limited documentation** — improving but gaps remain
- **Small community** — limited support resources

## When to Consider C3

- **Exploring C alternatives** — if you want C's model with modern safety features
- **Learning language design** — C3 is an interesting case study in conservative evolution
- **Small embedded projects** — where C's UB is a concern but Rust/Zig feel too different
- **Gradual C migration** — C ABI compatibility allows incremental adoption
- **Not recommended for production** — the language is still experimental
